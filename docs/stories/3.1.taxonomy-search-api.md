# Story 3.1: Taxonomy Search API

## Status
Done

## Story
**As a** Mobile Client,
**I want** to query the local database for bird taxonomy by common or scientific name,
**so that** I can quickly auto-complete species searches without hitting external APIs.

## Acceptance Criteria
1. A `GET /api/v1/taxonomy/search?q={query}` endpoint is created.
2. The endpoint queries the `bird_taxonomy` table for records where `common_name` or `scientific_name` matches the query (case-insensitive, partial matches allowed).
3. The endpoint returns a maximum of 15 results to ensure fast response times.
4. The response format matches the expected `BirdSpecies` payload (species_code, common_name, scientific_name).
5. The endpoint is secured with JWT authentication.
6. A test verifies that the custom JPA repository query correctly returns partial, case-insensitive matches.

## Dev Notes
- **Database Query Performance**: Decided to utilize a native PostgreSQL query utilizing `ILIKE` matching to respect the existing indexing strategy while strictly enforcing a `LIMIT 15` directly at the database layer (preventing unnecessary data hydration of up to 17,000 birds into Java memory).
- **Security Check**: The controller relies on the global `JwtAuthFilter` configured previously, ensuring no manual auth checks pollute the controller tier.

### Testing
- Fully integrated test utilizing Testcontainers spins up a real PostgreSQL instance, creating authentic DB boundaries for the `ILIKE` queries and validating the hard limit works securely alongside JWT.
