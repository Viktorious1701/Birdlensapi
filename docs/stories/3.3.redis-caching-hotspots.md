# Story 3.3: Redis Caching for Nearby Hotspots

## Status
Done

## Story
**As a** Backend Developer,
**I want** to cache the results of the "nearby hotspots" endpoint in Redis,
**so that** repeated queries for the same popular locations return in under 50ms and reduce database load.

## Acceptance Criteria
1. Spring Data Redis and `spring-boot-starter-cache` are integrated into the project.
2. The `@Cacheable` annotation (or manual `RedisTemplate` logic) is applied to the `hotspots/nearby` endpoint service method.
3. The cache key is generated by rounding the `lat` and `lng` to a specific precision (e.g., 2 decimal places) and including the `radiusKm` to ensure high cache hit rates for users in the same general area.
4. The cache TTL (Time To Live) is configured globally or specifically for this cache to a reasonable duration (e.g., 1 hour) to balance freshness with performance.
5. A Testcontainers-backed Redis integration test verifies that the first call hits the database (using `@SpyBean` to count repository invocations) and subsequent identical calls hit the cache without invoking the repository.

## Dev Notes
- **SpEL Power Routing**: Utilized a dynamic Spring Expression Language query to natively round math expressions inside the annotation itself: `key = "T(java.lang.Math).round(#lat * 100.0) / 100.0 + ':' + T(java.lang.Math).round(#lng * 100.0) / 100.0 + ':' + #radiusKm"`. This prevents creating messy boilerplate keys in the controller.
- **Record Serialization**: Java 14+ Records combined with `java.time.LocalDate` requires Jackson's `JavaTimeModule` and strict Polymorphic Type definitions enabled. We mapped a custom `ObjectMapper` straight into the `GenericJackson2JsonRedisSerializer` to handle this efficiently.

### Testing
- Integration tests confirm both identical hits and "geographically close" hits (due to rounding bounds) successfully bypass PostgreSQL after the first execution.
