# Story 3.3: Redis Caching for Nearby Hotspots

## Status
Draft

## Story
**As a** Backend Developer,
**I want** to cache the results of the "nearby hotspots" endpoint in Redis,
**so that** repeated queries for the same popular locations return in under 50ms and reduce database load.

## Acceptance Criteria
1. Spring Data Redis and `spring-boot-starter-cache` are integrated into the project.
2. The `@Cacheable` annotation (or manual `RedisTemplate` logic) is applied to the `hotspots/nearby` endpoint service method.
3. The cache key is generated by rounding the `lat` and `lng` to a specific precision (e.g., 2 decimal places) and including the `radiusKm` to ensure high cache hit rates for users in the same general area.
4. The cache TTL (Time To Live) is configured globally or specifically for this cache to a reasonable duration (e.g., 1 hour) to balance freshness with performance.
5. A Testcontainers-backed Redis integration test verifies that the first call hits the database (using `@SpyBean` to count repository invocations) and subsequent identical calls hit the cache without invoking the repository.

## Tasks / Subtasks
- [ ] Task 1: Add Dependencies (AC: 1)
  - [ ] Add `spring-boot-starter-data-redis` and `spring-boot-starter-cache` to `build.gradle`.
- [ ] Task 2: Configure Redis and CacheManager (AC: 1, 4)

  - [ ] Create a `RedisCacheManager` bean that configures a default TTL (e.g., 1 hour) and uses JSON serialization for the cache values (so we can easily inspect the cache via Redis CLI).

  - [ ] Update `HotspotService.findNearbyHotspots`.

  - [ ] Ensure the cache name is something clear, like `hotspots`.
- [ ] Task 4: Integration Testing (AC: 5)
  - [ ] Update `AbstractIntegrationTest` if necessary to ensure the Redis Testcontainer is fully mapped. (It should be mapped to `spring.data.redis.host` and `port`).
  - [ ] Create or update `HotspotIntegrationTest`.

## Dev Notes
- **Cache Key Design**: If a user is at `10.781` and another is at `10.784`, querying the database for a 10km radius will yield practically identical results. By rounding both to `10.78` before caching, both users share the same cache key (`hotspots:10.78:106.70:10.0`), massively increasing our cache hit ratio.
- **Key Generation Snippet**: 
  You can round the coordinates inside the service layer before calling a private/internal `@Cacheable` method, or build a custom `KeyGenerator` bean. Rounding to 2 decimal places provides roughly a 1km grid resolution.
  `double roundedLat = Math.round(lat * 100.0) / 100.0;`
- **Serialization**: By default, Spring Cache uses Java native serialization, which is unreadable in Redis and tightly couples to class definitions. Configure the `RedisCacheConfiguration` to use `GenericJackson2JsonRedisSerializer`.

### Testing
- Rely heavily on `@SpyBean` for your tests. It's the only way to definitively prove that a method execution was bypassed by the Spring Cache interceptor.
