# Story 2.4: Scheduled Spatial Hotspot Ingestion Job

## Status
Draft

## Story
**As the** System,
**I want** a scheduled background job to ingest notable birding hotspots per region,
**so that** the mobile app can query them locally without hitting eBird.

## Acceptance Criteria
1. A Spring `@Service` (`HotspotIngestionJob`) containing an `@Scheduled` method runs periodically (e.g., every 6 hours).
2. The service iterates through a configured list of supported regions (e.g., `["US", "VN", "GB"]` loaded from `application.yml`).
3. For each region, the service calls the eBird `v2/data/obs/{regionCode}/recent/notable` endpoint via the `EbirdApiClient`.
4. The service maps the observations to unique `ebird_hotspots` records. (Note: Multiple observations may occur at the same hotspot; the job must group/deduplicate them by `locId` before inserting).
5. The service performs an "Upsert" using `JdbcTemplate.batchUpdate()` against the PostGIS database.
6. The service correctly maps the API's `lat` and `lng` floats into the PostGIS `GEOGRAPHY(Point, 4326)` column using the `ST_SetSRID(ST_MakePoint(?, ?), 4326)` SQL function.
7. The service updates the `latest_obs_dt` and increments/updates `num_species_all_time` for existing hotspots, and inserts new hotspots.
8. The service logs its progress and failures for each region processed.

  - [ ] Add `fetchRecentNotableObservations(String regionCode)` to `EbirdApiClient`.
  - [ ] Create `EbirdObservationDto` to map the API response (`locId`, `locName`, `lat`, `lng`, `obsDt`).
- [ ] Task 2: Build the Ingestion Job (AC: 1, 2, 4)
  - [ ] Create `HotspotIngestionJob.java` in `com.example.birdlensapi.ingestion`.
  - [ ] Inject the list of supported regions from application properties using `@Value`.
  - [ ] Configure `@Scheduled` (e.g., `cron = "0 0 */6 * * *"` for every 6 hours).
  - [ ] Implement deduplication logic: The API returns *observations* (birds seen), but we are saving *hotspots* (locations). Group the returned list by `locId`, taking the most recent `obsDt` for each hotspot.
- [ ] Task 3: Implement PostGIS Batch Upsert (AC: 5, 6, 7, 8)
  - [ ] Inject `JdbcTemplate`.
  - [ ] Write the UPSERT query. Crucially, the `location` column must be populated using `ST_SetSRID(ST_MakePoint(?, ?), 4326)::geography` in the `VALUES` clause, passing `longitude` then `latitude`.
  - [ ] Wrap the region loop in a try-catch so one failing region does not stop the others.

## Dev Notes
- **Deduplication**: eBird's notable sightings API returns a list of individual bird sightings. If 5 rare birds are seen at Central Park, the API returns 5 objects with the same `locId` ("L12345"). You *must* deduplicate this list by `locId` in Java before passing it to `JdbcTemplate` to prevent unique constraint violations within the same transaction batch.
- **PostGIS Syntax**: The exact SQL for the location column insert is `ST_SetSRID(ST_MakePoint(?, ?), 4326)::geography`. Note that `ST_MakePoint` takes `longitude` first, then `latitude` (X, Y).
- **Upsert SQL Example**:
  ```sql
  INSERT INTO ebird_hotspots (loc_id, loc_name, latitude, longitude, location, latest_obs_dt)
  VALUES (?, ?, ?, ?, ST_SetSRID(ST_MakePoint(?, ?), 4326)::geography, ?)
  ON CONFLICT (loc_id) DO UPDATE SET
      loc_name = EXCLUDED.loc_name,
      latitude = EXCLUDED.latitude,
      longitude = EXCLUDED.longitude,
      location = EXCLUDED.location,
      latest_obs_dt = GREATEST(ebird_hotspots.latest_obs_dt, EXCLUDED.latest_obs_dt)
  ```

### Testing
- Unit test the deduplication logic to ensure 10 observations at the same location resolve to exactly 1 hotspot DTO with the most recent date.
- If possible, add a test to ensure the `ST_MakePoint` parameters are passed as `(longitude, latitude)` and not reversed.
