# Story 1.4: JWT-based Authentication (Login)

## Status
Draft

## Story
**As a** Mobile User,
**I want** to log in with my email and password to receive a secure session token (JWT)
**so that** I can authenticate my future API requests.

## Acceptance Criteria
1. Spring Security is configured to support stateless, JWT-based authentication.
2. A `POST /api/v1/auth/login` endpoint accepts `email` and `password`.
3. Upon successful credential verification, the endpoint returns a `200 OK` containing a short-lived `access_token` (JWT) and a long-lived `refresh_token`.
4. The `access_token` payload includes the user's ID and role.
5. Invalid credentials return a `401 Unauthorized` status.
6. A `JwtAuthFilter` is implemented to extract and validate tokens from the `Authorization: Bearer` header on subsequent requests.

## Tasks / Subtasks
- [ ] Task 1: Implement JWT Utility (AC: 3, 4)
  - [ ] Add `jjwt-api`, `jjwt-impl`, and `jjwt-jackson` to `build.gradle` (Version 0.12.x per architecture).
  - [ ] Create `JwtService` to handle token generation, extraction, and validation.
  - [ ] Ensure the token includes `userId` and `roles` as claims.
- [ ] Task 2: Create Login DTOs (AC: 2, 3)
  - [ ] Create `LoginRequest` (email, password).
  - [ ] Create `LoginResponse` (accessToken, refreshToken).
- [ ] Task 3: Implement Login Logic (AC: 2, 3, 5)
  - [ ] Add `login` method to `UserService`.
  - [ ] Use `AuthenticationManager` to verify credentials.
  - [ ] Return `LoginResponse` on success; Spring Security will handle 401 on failure if configured correctly.
- [ ] Task 4: Implement JWT Security Filter (AC: 6)
  - [ ] Create `JwtAuthFilter` extending `OncePerRequestFilter`.
  - [ ] Configure it to look for `Bearer` tokens in headers.
  - [ ] If valid, set the `SecurityContextHolder`.
- [ ] Task 5: Finalize Security Configuration (AC: 1, 6)
  - [ ] Update `SecurityConfig` to add `JwtAuthFilter` before the `UsernamePasswordAuthenticationFilter`.
  - [ ] Set Session Creation Policy to `STATELESS`.
  - [ ] Define `AuthenticationManager` bean.
- [ ] Task 6: Integration Testing (AC: 5)
  - [ ] Update `AuthIntegrationTest` to verify the login flow.
  - [ ] Verify that a protected endpoint (like `/api/v1/users/me`—planned for next story) returns 401 without a token and 200 with a valid token.

## Dev Notes
- **JWT Version**: Use `io.jsonwebtoken:jjwt:0.12.x`. Note that the API changed significantly in 0.12.0 (e.g., using `Jwts.builder().content(...)` vs `setClaims`).
- **Statelessness**: It is vital that we do **not** use `HttpSession`. All security context must be rebuilt from the JWT on every request.
- **Refresh Tokens**: For the MVP, the `refresh_token` can just be another JWT with a longer expiry, or a random UUID stored in the database. For now, a longer-lived JWT is simpler to implement.
- **Environment Variables**: Ensure `JWT_SECRET` is pulled from environment variables and not hardcoded.

### Testing
- `POST http://localhost:8088/api/v1/auth/login`
- Body: `{"email": "test@example.com", "password": "password123"}`
- Expect: `{"success": true, "data": {"accessToken": "...", "refreshToken": "..."}, ...}`
