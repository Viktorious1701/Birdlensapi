# Story 2.2: Configure Resilient eBird API Client

## Status
Done

## Story
**As the** System,
**I want** a resilient external API client configured for eBird,
**so that** I can fetch taxonomy and hotspot data, handling timeouts, retries, and rate limits gracefully.

## Acceptance Criteria
1. A Spring `@Configuration` class defines a `WebClient` specifically for `https://api.ebird.org/v2/`.
2. The client is configured with sensible connection timeouts (e.g., 5 seconds) and read timeouts (e.g., 10 seconds).
3. A retry mechanism using Resilience4j is configured to automatically retry `5xx` server errors or network timeouts up to 3 times with an exponential backoff (multiplier 2).
4. An Interceptor or `ExchangeFilterFunction` automatically injects the eBird API key (configured via `app.ebird.api-key` property) into the `x-ebirdapitoken` header for every request.
5. A MockWebServer-backed integration test proves that a `200 OK` response parses correctly, and a simulated `503 Service Unavailable` triggers the retry mechanism before failing gracefully.

## Dev Notes
- **WebClient vs WebMVC**: Configured safely using the Netty `HttpClient` under the hood. Since the main app remains in Servlet mode, reactive returns are fine but will eventually be `.block()`'d locally inside scheduled jobs later on.
- **Resilience4j AOP**: Added `spring-boot-starter-aop`. `@Retry` annotations are intercepted by AOP proxies. Without the AOP dependency, the annotation is completely ignored by Spring Boot 3.

### Testing
- `EbirdApiClientIntegrationTest` bypasses actual eBird servers via `MockWebServer`. Tested 200 OKs, 503 recovery (after 2 tries), and maximum exhaustion (after 3 tries throwing `WebClientResponseException.ServiceUnavailable`).
