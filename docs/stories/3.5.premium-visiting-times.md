# Story 3.5: Premium Visiting Times API

## Status
Done

## Story
**As a** Premium Mobile Client,
**I want** to fetch the analytical visiting times for a hotspot,
**so that** I can see the best months and hours to visit.

## Acceptance Criteria
1. A `GET /api/v1/hotspots/{locId}/visiting-times` endpoint is created.
2. The endpoint queries an analytical view or aggregation of historical sightings for the specified `locId`. (Note: Since we are not ingesting full historical observations in this MVP, return a mocked/calculated `VisitingTimesAnalysis` response based on the `num_species_all_time` or the `latest_obs_dt` to satisfy the API contract).
3. The endpoint validates the user's JWT and their database profile to ensure they have an active premium subscription (e.g., `subscription_id` is not null and `subscription_expires_at` is in the future).
4. If the user does not have an active subscription, the endpoint returns a `403 Forbidden` status.
5. The response format matches the `VisitingTimesAnalysis` model expected by the Android app (Monthly and Hourly stats).
6. A test verifies that a standard user receives a 403, and a premium user receives a 200 OK.

## Dev Notes
- **Authorization Implementation**: Handled directly via database validation within `HotspotService`. `AccessDeniedException` flows perfectly through to our global exception handler to provide a structured `403 Forbidden` response.
- **Deterministic Mocking**: Utilized `java.util.Random` seeded specifically with the hashcode of `locId`. This guarantees the exact same visual analytical chart layout is returned every single time a user queries a specific hotspot, establishing a consistent UX during testing.

### Testing
- Appended `shouldReturn403ForbiddenWhenStandardUserRequestsVisitingTimes`, `shouldReturnVisitingTimesWhenPremiumUserRequests`, and `shouldReturn404WhenPremiumUserRequestsVisitingTimesForNonExistentHotspot` to ensure full boundary condition compliance within `HotspotIntegrationTest`.
