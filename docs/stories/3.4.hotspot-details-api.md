# Story 3.4: Fetch Hotspot Details API

## Status
Done

## Story
**As a** Mobile Client,
**I want** to fetch detailed information about a specific hotspot ID,
**so that** I can view its recent sightings and statistics.

## Acceptance Criteria
1. A `GET /api/v1/hotspots/{locId}` endpoint is created.
2. The endpoint queries the `ebird_hotspots` table by its unique `loc_id`.
3. If the hotspot is not found in the local database, the endpoint returns a `404 Not Found`.
4. The response format matches the `EbirdNearbyHotspot` model expected by the Android app.
5. The endpoint is secured with JWT authentication.
6. An integration test verifies both a successful fetch (200 OK) and a missing fetch (404 Not Found).

## Dev Notes
- **Primary Key Fetching**: Spring Data JPA provides `findById` natively, skipping the need to write an explicit JPQL/native query.
- **Exception Handling Mapping**: The `ResourceNotFoundException` intercepts perfectly with the `GlobalExceptionHandler` built previously, automatically transforming the error into the standardized `ApiResponse` shape.
- **Caching**: Implemented an explicit `@Cacheable(value = "hotspot_details", key = "#locId")` to cache direct lookups using the same underlying Redis Manager logic.

### Testing
- Using the existing `HotspotIntegrationTest` suite, I added `shouldReturnHotspotDetailsSuccessfully` and `shouldReturn404ForNonExistentHotspot`. Both accurately leverage a dynamically generated JWT token to satisfy security constraints.
