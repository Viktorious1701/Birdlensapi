# Story 2.1: Define Bird Taxonomy and Spatial Hotspot Schemas

## Status
Draft

## Story
**As a** Backend Developer,
**I want** to define the core database schemas for Bird Taxonomy and Spatial Hotspots,
**so that** ingested eBird data can be queried efficiently by the mobile application.

## Acceptance Criteria
1. A Flyway migration script creates a `bird_taxonomy` table (species_code, common_name, scientific_name, category, taxon_order, bird_order, family_common_name, family_scientific_name).
2. A Flyway migration script creates an `ebird_hotspots` table (loc_id, loc_name, country_code, subnational1_code, subnational2_code, latitude, longitude, latest_obs_dt, num_species_all_time).
3. The `latitude` and `longitude` columns are mapped to a PostGIS `GEOGRAPHY(Point, 4326)` column named `location` for efficient spatial querying.
4. Appropriate indexes (e.g., a GiST index on the `location` column) are created to optimize bounding box and radius queries.
5. Spring Data JPA entities are created and accurately mapped to these tables.

## Tasks / Subtasks
- [ ] Task 1: Create Flyway Migration (AC: 1, 2, 3, 4)
  - [ ] Create `V2__init_taxonomy_hotspots.sql` in `src/main/resources/db/migration/`.
  - [ ] Define the `bird_taxonomy` table and add B-Tree indexes on `common_name` and `scientific_name` using `text_pattern_ops` (per architecture docs).
  - [ ] Define the `ebird_hotspots` table including the `GEOGRAPHY(Point, 4326)` column.
  - [ ] Create a GiST index on the `location` column of `ebird_hotspots`.
- [ ] Task 2: Create JPA Entities (AC: 5)
  - [ ] Add Hibernate Spatial dependency to `build.gradle` if not already present.
  - [ ] Create `BirdTaxonomy` entity in `com.example.birdlensapi.domain.taxonomy` with `@Id` mapped to `species_code`.
  - [ ] Create `EbirdHotspot` entity in `com.example.birdlensapi.domain.hotspot` with `@Id` mapped to `loc_id`.
  - [ ] Map the spatial column in `EbirdHotspot` using `@Column(columnDefinition = "GEOGRAPHY(Point, 4326)")` and the `org.locationtech.jts.geom.Point` type.
- [ ] Task 3: Create Repositories (AC: 5)
  - [ ] Create `TaxonomyRepository` extending `JpaRepository`.
  - [ ] Create `HotspotRepository` extending `JpaRepository`.

## Dev Notes
- **Architecture Reference**: The exact SQL definitions for these tables are located in `docs/architecture/data-models.md`. Use those precise definitions to ensure the GiST and text pattern indexes are created correctly.
- **Dependencies**: To work with PostGIS types in Java, you will likely need to add `org.hibernate.orm:hibernate-spatial` and `org.locationtech.jts:jts-core` to the `build.gradle`.
- **Entity IDs**: Notice that these tables do *not* use UUIDs. `bird_taxonomy` uses `species_code` (VARCHAR) as its primary key, and `ebird_hotspots` uses `loc_id` (VARCHAR). Do not use `@GeneratedValue` on these entities, as the IDs will be provided by the eBird API during ingestion.

### Testing
- Run the Spring Boot application locally to ensure Flyway successfully applies `V2__init_taxonomy_hotspots.sql`.
- Verify the context loads successfully and JPA validates the schema (`ddl-auto: validate` is active).
