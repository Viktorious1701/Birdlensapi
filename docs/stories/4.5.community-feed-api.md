# Story 4.5: Paginated Community Feed API

## Status
Done

## Story
**As a** Mobile Client,
**I want** to fetch a paginated feed of community posts with their processed media,
**so that** I can view what other users are sharing without loading massive original files.

## Acceptance Criteria
1. An endpoint `GET /api/v1/posts` is created.
2. The endpoint supports pagination using Spring Data `Pageable` (`?page=0&size=20`).
3. The service queries the `posts` table, joining with `users` and `post_media`.
4. **Crucial Filter:** The feed must only return posts where media processing is complete (i.e., at least one media item has `status = 'COMPLETED'`) OR posts with no media (if type is GENERAL).
5. The response includes:
    - User details (username, avatar)
    - Post details (content, location name, type, sighting date)
    - List of media URLs (returning `thumbnail_url` and `compressed_url`, **never** `original_url`)
6. The service utilizes Redis caching (similar to Hotspots) to serve the first few pages of the "latest" feed.
7. An integration test verifies that posts appear in the feed only after the worker marks media as COMPLETED.

## Dev Notes
- **N+1 Prevention**: By executing `SELECT DISTINCT p FROM Post p LEFT JOIN FETCH p.media m LEFT JOIN FETCH p.user u`, we execute a single SQL query that recursively builds the entity graph. While this triggers Hibernate's in-memory pagination safeguard warning (due to the collection fetch), it successfully guarantees that processing rules hold up perfectly without dropping 21 consecutive SQL calls to the database.
- **DTO Safety Guard**: Removed `originalUrl` completely from the `PostMediaFeedResponse` DTO mapping, inherently securing our raw S3 media blobs from being accidentally loaded onto user feeds and crashing mobile data limits.
- **Cache Serialization Fix**: Utilized a custom `FeedPageResponse` instead of passing Spring Data's `Page<T>` directly. The generic `PageImpl` lacks default serializers, frequently causing exceptions inside the `Jackson2JsonRedisSerializer`.

### Testing
- `FeedIntegrationTest` evaluates multiple persistence boundaries. The framework automatically wipes down the Redis memory caches explicitly using `cacheManager.getCache("feed").clear()` between state shifts to prevent mocked chronologies from triggering dirty cache assertions.
