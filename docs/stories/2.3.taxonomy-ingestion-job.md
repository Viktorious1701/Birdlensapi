# Story 2.3: Scheduled eBird Taxonomy Ingestion Job

## Status
Done

## Story
**As the** System,
**I want** a scheduled background job to ingest the full eBird Taxonomy,
**so that** the local database has an up-to-date, searchable list of all bird species.

## Acceptance Criteria
1. A Spring `@Service` (`TaxonomyIngestionJob`) containing an `@Scheduled` method is created to run periodically (e.g., once a week, or on application startup if the table is empty).
2. The service calls the eBird `v2/ref/taxonomy/ebird?fmt=json` endpoint via the `EbirdApiClient` and parses the massive JSON response into a list of DTOs.
3. The service processes the payload efficiently by partitioning the list into batches of 500.
4. The service performs an "Upsert" (Update if exists, Insert if new) based on the unique `species_code` using `JdbcTemplate.batchUpdate()`.
5. The ingestion process logs its start time, the number of records processed, any errors encountered, and its completion time.
6. The job gracefully handles partial failures: if one batch fails, it logs the error and continues processing the remaining batches (best-effort).

## Dev Notes
- **Architecture Reference**: Bypassed Spring Data JPA `saveAll()` as strictly requested, utilizing low-level `JdbcTemplate.batchUpdate()` paired with raw `ON CONFLICT` constraints to provide native PostgreSQL UPSERT operations efficiently.
- **Execution Strategy**: Instead of `@PostConstruct`, which would block Tomcat startup, used `@EventListener(ApplicationReadyEvent.class)`. This ensures the database is checked and the initial sync is triggered *after* the application is fully running and ready to accept traffic.

### Testing
- `TaxonomyIngestionJobTest.java` was written using Mockito. It fakes a 1000-item return list from eBird, verifies it slices exactly into two batches of 500, forces the first batch to throw a Database exception, and asserts that the code swallows the error and proceeds to execute the second batch successfully.
