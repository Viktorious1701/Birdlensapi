# Story 1.2: Configure Flyway for database migrations

## Status
Done

## Story
**As a** Backend Developer,
**I want** to configure Flyway for database migrations
**so that** schema changes are reliably tracked, versioned, and applied automatically on application startup.

## Acceptance Criteria
1. Flyway dependencies are integrated into the `build.gradle`.
2. An initial migration script (`V1__init_users.sql`) creates the core `users` table (id, email, password_hash, username, first_name, last_name, avatar_url, subscription_id, subscription_expires_at, created_at, updated_at).
3. The application startup logs confirm that Flyway successfully applied the migration to the Dockerized PostgreSQL database.

## Dev Notes
- **Architecture Reference**:
  - The `users` table schema is defined in the Architecture document under "Core Tables". Use that exact SQL.
  - **CRITICAL CONSTRAINT**: The Architecture document shows `users` having a foreign key: `subscription_id UUID REFERENCES subscriptions(id)`. Since the `subscriptions` table does not exist yet (it is planned for Epic 5), you have two choices to make this migration work now:
    1.  **Recommended approach**: Create a minimal `subscriptions` table in `V1__init_users.sql` *before* the `users` table. The Architecture document states: `subscriptions (standalone lookup table)`. A simple table with `id UUID PRIMARY KEY DEFAULT gen_random_uuid()` and a `name VARCHAR` will suffice for now to satisfy the FK constraint.
    2.  Alternatively, remove the `REFERENCES subscriptions(id)` constraint from the `users` table in V1, and add it later via an `ALTER TABLE` in a V5 migration.
  - Ensure the `id` column uses `UUID PRIMARY KEY DEFAULT gen_random_uuid()`.
  - All migration filenames must follow the pattern: `V{n}__{description_in_snake_case}.sql` (Note the double underscore).

### Testing
- Run `docker-compose up -d postgres` if the database is not already running.
- Start the application. Look for logs similar to: `Successfully validated 1 migration` and `Successfully applied 1 migration to schema "public" (execution time ... )`.
- Inspect the database schema to confirm the tables were created successfully.
