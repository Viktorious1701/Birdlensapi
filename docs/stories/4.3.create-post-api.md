# Story 4.3: Create Post API & Event Publishing

## Status
Done

## Story
**As a** Mobile Client,
**I want** to create a new "Sighting" post with text and media references, receiving an immediate successful response,
**so that** I am not blocked while media processing happens asynchronously.

## Acceptance Criteria
1. An endpoint `POST /api/v1/posts` accepts post metadata (content, location, etc.) and a list of object storage keys/URLs corresponding to the uploaded files.
2. The service creates a new record in the `posts` table.
3. The service creates records in the `post_media` table with a `processing_status` of `PENDING`.
4. The service publishes a `PostCreatedEvent` message to a RabbitMQ exchange (`posts.exchange`) with the `post_id`.
5. The endpoint immediately returns a `201 Created` with the new post's data to the client.
6. A Testcontainers-backed integration test verifies that the database records are created and a message is successfully published to RabbitMQ.

## Dev Notes
- **Spatial Parsing**: Parsed raw lat/lng values strictly using JTS `GeometryFactory`, ensuring PostGIS compliance during save operations.
- **Message Serialization**: Configured RabbitMQ to enforce JSON serialization for events. This allows our backend queues to remain human-readable when inspected via the RabbitMQ UI panel locally, avoiding obscure binary blobs.

### Testing
