# Story 3.2: Nearby Hotspots API (Spatial Query)

## Status
Done

## Story
**As a** Mobile Client,
**I want** to query the local database for nearby birding hotspots using a latitude, longitude, and radius,
**so that** the map loads quickly without relying on eBird.

## Acceptance Criteria
1. A `GET /api/v1/hotspots/nearby?lat={lat}&lng={lng}&radiusKm={radiusKm}` endpoint is created.
2. The endpoint utilizes a PostGIS spatial query (e.g., `ST_DWithin`) against the `location` column of the `ebird_hotspots` table.
3. The endpoint returns a list of hotspots within the specified radius, ordered by `num_species_all_time` descending.
4. The endpoint limits the response to 50 hotspots to ensure fast response times and prevent map clustering overhead.
5. The response format matches the `EbirdNearbyHotspot` model expected by the Android app.
6. The endpoint is secured with JWT authentication.
7. A Testcontainers-backed integration test validates that points inside the radius are included, and points outside the radius are correctly excluded from the response.

## Dev Notes
- **Radius Conversion**: The client provides `radiusKm` (kilometers), but `ST_DWithin` for `GEOGRAPHY` types natively expects distance in **meters**. Logic was added to the `HotspotService` to multiply `radiusKm * 1000.0`.
- **Validation Handlers**: Using `@Validated` at the controller class level triggers `ConstraintViolationException`. Added a handler in `GlobalExceptionHandler` so Spring correctly returns `400 Bad Request` instead of crashing to a `500 Server Error`.

### Testing
- `HotspotIntegrationTest` successfully populates three spatial coordinates utilizing the `org.locationtech.jts.geom.GeometryFactory` with SRID 4326. Assertions prove exact radius inclusions, exclusions, descending sort boundaries, and coordinate validation constraints.
